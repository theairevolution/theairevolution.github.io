REST is the best communication strategy for your Python app — or is it the only one you know?

A practical deep-dive into 9 communication patterns every Python developer should have in their toolbox — with code examples, architectural diagrams, and honest trade-offs.

00 · Introduction

Most APIs are REST. Most problems aren't.

Ask any Python developer how their service talks to the outside world and nine times out of ten you'll hear "REST". It's the default. The safe bet. The answer on every tutorial. And it's a great tool — don't get me wrong.

But REST is just one communication pattern, and the problem with defaults is that we stop asking whether they're the right choice. A hammer doesn't mean every problem is a nail, and an HTTP request-response cycle doesn't mean every data exchange is a synchronous fetch.

In this article I'll walk through 9 communication strategies available in Python, from the classics you already know to the patterns that will completely change how you architect your next system. Each one has a diagram, a working code example, and an honest breakdown of when to use it — and when not to.

🔗 Code Reference

The HTTP Streaming, SSE, and WebSocket examples are directly based on the communication_methods_python repo, built with FastAPI. The other patterns (gRPC, raw sockets, message queues, UDP, long polling) are original extensions following the same structure.

02 · Pattern One

REST / HTTP

HTTP/1.1

REST (Representational State Transfer) is the architecture style that most of the web runs on. Each request carries all the context needed, the server processes it, and responds. Simple, stateless, and battle-tested.

// REST — Request / Response cycle

FastAPI REST Example

rest/server.py Python 
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

# In-memory store (replace with your DB of choice)
items: dict = {}

class Item(BaseModel):
    name: str
    price: float

@app.get("/items/{item_id}")
async def get_item(item_id: int):
    if item_id not in items:
        raise HTTPException(status_code=404, detail="Not found")
    return items[item_id]

@app.post("/items/{item_id}", status_code=201)
async def create_item(item_id: int, item: Item):
    items[item_id] = item
    return {"id": item_id, "item": item}
✅ Best Use Cases

CRUD APIs, public-facing endpoints, microservice-to-microservice calls where simplicity matters, anything that benefits from HTTP caching, mobile backends, third-party integrations.

⚠️ When to Reconsider

When you need sub-100ms push notifications, bidirectional streaming, high-frequency updates (>10/sec), or decoupled async processing. REST will work — but it'll be inefficient.

03 · Pattern Two

WebSocket

Full-Duplex TCP

WebSocket upgrades a regular HTTP connection into a persistent, bidirectional channel. Once the handshake is done, both the client and server can send messages to each other at any time with minimal overhead — no need to re-establish a connection per message.

// WebSocket — persistent bidirectional channel

FastAPI WebSocket Server

web_socket/server.py Python 
from fastapi import FastAPI, WebSocket, WebSocketDisconnect

app = FastAPI()

# Track connected clients
connected_clients: list[WebSocket] = []

@app.websocket("/ws")
async def websocket_endpoint(ws: WebSocket):
    await ws.accept()
    connected_clients.append(ws)
    print(f"Client connected. Total: {len(connected_clients)}")

    try:
        while True:
            data = await ws.receive_text()
            # Echo back + broadcast to all clients
            for client in connected_clients:
                await client.send_text(f"[broadcast] {data}")
    except WebSocketDisconnect:
        connected_clients.remove(ws)
        print("Client disconnected")

Python WebSocket Client

web_socket/client.py Python 
import asyncio
import websockets

async def run():
    async with websockets.connect("ws://localhost:8002/ws") as ws:
        await ws.send("Hello from client!")
        response = await ws.recv()
        print(f"Received: {response}")

asyncio.run(run())
✅ Best Use Cases

Real-time chat, collaborative editors (Google Docs style), live dashboards with bidirectional control, multiplayer games, financial trading UIs, pair programming tools.

04 · Pattern Three

Server-Sent Events (SSE)

HTTP Push

SSE is WebSocket's simpler cousin — a long-lived HTTP connection where only the server sends data. The client subscribes once and listens. It works over plain HTTP, auto-reconnects on failure, and is natively supported by browsers without any library.

// SSE — one-way server push

FastAPI SSE Server

sse/server.py Python 
import asyncio
from fastapi import FastAPI
from sse_starlette.sse import EventSourceResponse

app = FastAPI()

async def event_generator():
    for i in range(10):
        yield {
            "event": "update",
            "data": f"Server time tick #{i}",
        }
        await asyncio.sleep(1)

@app.get("/events")
async def sse_endpoint():
    return EventSourceResponse(event_generator())
✅ Best Use Cases

Live notifications, activity feeds, CI/CD build logs, AI-generated text streaming (like ChatGPT responses!), stock tickers, sports scores — any scenario where the server continuously pushes data to the browser.

05 · Pattern Four

HTTP Streaming

Chunked Transfer

HTTP Streaming uses chunked transfer encoding to send a large response progressively, in parts. The server doesn't need to know the total size upfront — it writes chunks as they become available. Unlike SSE, this is a single response body being delivered in pieces, not a stream of events.

FastAPI Streaming Response

stream_http/server.py Python 
import asyncio
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

async def generate_large_content():
    # Simulate an LLM or file being streamed token-by-token
    words = ["The ", "quick ", "brown ", "fox ", "jumps..."]
    for word in words:
        yield word.encode()
        await asyncio.sleep(0.3)

@app.get("/stream")
async def stream_endpoint():
    return StreamingResponse(
        generate_large_content(),
        media_type="text/plain"
    )
✅ Best Use Cases

LLM token streaming (OpenAI / Anthropic style), large file downloads, progressive image loading, report generation where results are computed and delivered incrementally.

06 · Pattern Five

gRPC

HTTP/2 + Protobuf

gRPC is Google's high-performance RPC framework. It uses Protocol Buffers for binary serialization (10x more compact than JSON) and HTTP/2 for multiplexed, low-latency transport. You define your service contract in a .proto file and gRPC generates client/server stubs.

// gRPC — contract-first service communication

Proto Definition + Python Server

grpc/user.proto Protobuf 
syntax = "proto3";

service UserService {
  rpc GetUser (UserRequest) returns (UserResponse);
  rpc StreamUsers (UserRequest) returns (stream UserResponse);
}

message UserRequest { int32 id = 1; }
message UserResponse {
  int32  id    = 1;
  string name  = 2;
  string email = 3;
}
grpc/server.py Python 
import grpc
from concurrent import futures
import user_pb2, user_pb2_grpc

class UserServicer(user_pb2_grpc.UserServiceServicer):
    def GetUser(self, request, context):
        return user_pb2.UserResponse(
            id=request.id,
            name=f"User {request.id}",
            email=f"user{request.id}@example.com"
        )

    def StreamUsers(self, request, context):
        for i in range(5):
            yield user_pb2.UserResponse(id=i, name=f"User {i}")

server = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
user_pb2_grpc.add_UserServiceServicer_to_server(UserServicer(), server)
server.add_insecure_port("[::]:50051")
server.start()
server.wait_for_termination()
✅ Best Use Cases

Internal microservice communication (especially polyglot), high-throughput service meshes, mobile backends where bandwidth matters, bi-directional streaming between services, replacing REST in performance-critical internal APIs.

07 · Pattern Six

Message Queue

Async / Decoupled

Message queues are the backbone of resilient distributed systems. Instead of services calling each other directly, they publish messages to a broker (RabbitMQ, Kafka, SQS). Consumers process them independently — at their own pace, with retries, dead-letter queues, and durability guarantees.

// Message Queue — decoupled async pipeline

Python with aio-pika (RabbitMQ)

message_queue/producer.py Python 
import asyncio, json
import aio_pika

async def publish_order(order: dict):
    conn = await aio_pika.connect_robust("amqp://guest:guest@localhost/")
    async with conn:
        channel = await conn.channel()
        queue = await channel.declare_queue("orders", durable=True)

        await channel.default_exchange.publish(
            aio_pika.Message(
                body=json.dumps(order).encode(),
                delivery_mode=aio_pika.DeliveryMode.PERSISTENT,
            ),
            routing_key="orders",
        )
        print(f"Published: {order}")

asyncio.run(publish_order({"id": 42, "item": "keyboard"}))
message_queue/consumer.py Python 
async def consume_orders():
    conn = await aio_pika.connect_robust("amqp://guest:guest@localhost/")
    async with conn:
        channel = await conn.channel()
        queue = await channel.declare_queue("orders", durable=True)

        async with queue.iterator() as msgs:
            async for msg in msgs:
                async with msg.process():
                    order = json.loads(msg.body)
                    print(f"Processing order: {order}")
                    # ... business logic ...

asyncio.run(consume_orders())
✅ Best Use Cases

E-commerce order processing, email/notification dispatch, background job processing, event-driven microservices, fan-out scenarios (one event, multiple consumers), any system that needs retry logic and durability guarantees.

08 · Pattern Seven

UDP (User Datagram Protocol)

Connectionless

UDP is the speed demon of the networking world. It sends packets without establishing a connection, without ordering them, and without confirming delivery. That's not a bug — it's a feature. When you need the absolute lowest latency and can tolerate some loss, UDP is unmatched.

Python UDP Server & Client

udp/server.py Python 
import socket

def run_udp_server(host="0.0.0.0", port=9000):
    with socket.socket(socket.AF_INET, socket.SOCK_DGRAM) as sock:
        sock.bind((host, port))
        print(f"UDP server listening on {host}:{port}")

        while True:
            data, addr = sock.recvfrom(1024)
            print(f"[{addr}] {data.decode()}")
            # Fire-and-forget: no ACK, no guarantee

run_udp_server()
udp/client.py Python 
import socket, time

def send_telemetry():
    with socket.socket(socket.AF_INET, socket.SOCK_DGRAM) as sock:
        for i in range(100):
            payload = f"sensor:{i},temp:{22.5+i*0.1:.1f}"
            sock.sendto(payload.encode(), ("127.0.0.1", 9000))
            time.sleep(0.05)  # 20 packets/sec

send_telemetry()
✅ Best Use Cases

Online multiplayer games (position updates), video/audio streaming (VoIP, WebRTC internals), IoT sensor telemetry, DNS lookups, real-time monitoring where a dropped packet is acceptable.

09 · Pattern Eight

Raw TCP Sockets

TCP · OS Level

Raw TCP sockets give you maximum control. You define your own protocol over a persistent TCP connection — no HTTP overhead, no framework abstractions. You manage framing, serialization, and connection lifecycle yourself. It's the foundation that WebSocket, gRPC, and HTTP are all built on.

Python asyncio TCP Server

raw_socket/server.py Python 
import asyncio

async def handle_client(reader: asyncio.StreamReader,
                       writer: asyncio.StreamWriter):
    addr = writer.get_extra_info("peername")
    print(f"Connection from {addr}")

    try:
        while True:
            data = await reader.read(1024)
            if not data:
                break
            msg = data.decode().strip()
            print(f"[{addr}] → {msg}")
            writer.write(f"ACK: {msg}\n".encode())
            await writer.drain()
    finally:
        writer.close()

async def main():
    server = await asyncio.start_server(handle_client, "0.0.0.0", 8888)
    print("Raw TCP server on :8888")
    async with server:
        await server.serve_forever()

asyncio.run(main())
✅ Best Use Cases

Custom binary protocols, embedded systems, building your own protocol layer, proxy servers, high-frequency trading systems, game servers where every microsecond counts and HTTP overhead is not acceptable.

10 · Pattern Nine

Long Polling

HTTP · Held Open

Long polling is the oldest trick for simulating server push over plain HTTP. The client makes a request and the server holds it open until data is available (or a timeout), then responds. The client immediately makes a new request and waits again. It's polling, but smarter.

// Long Polling — held request → response → repeat

FastAPI Long Polling Endpoint

long_polling/server.py Python 
import asyncio
from fastapi import FastAPI

app = FastAPI()
pending_messages: list = []
new_message_event = asyncio.Event()

@app.get("/poll")
async def long_poll(timeout: float = 30.0):
    if pending_messages:
        return {"messages": pending_messages.pop(0)}

    try:
        # Block until a message arrives or timeout
        await asyncio.wait_for(new_message_event.wait(), timeout=timeout)
        new_message_event.clear()
        return {"messages": pending_messages.pop(0)}
    except asyncio.TimeoutError:
        return {"messages": None}  # client re-polls

@app.post("/push")
async def push_message(msg: str):
    pending_messages.append(msg)
    new_message_event.set()
✅ Best Use Cases

Legacy environments where WebSocket is blocked by proxies/firewalls, chat systems that need to work everywhere, gradual migration path from polling to real-time. In modern systems, prefer SSE or WebSocket — long polling is the compatibility fallback.

13 · Final Thoughts

The right tool changes everything

Every communication pattern in this article solves a real problem. The mistake isn't using REST — it's using REST when the problem calls for something else, and not knowing you had other options.

The good news: Python has first-class support for all of them. FastAPI makes SSE, WebSocket, and HTTP Streaming almost trivial. The standard library gives you raw sockets and UDP. grpcio, aio-pika, and confluent-kafka cover the heavier protocols with mature libraries.

Next time you're architecting a feature, ask yourself: is this a request or an event? Does the client need to push or just receive? Do I need guarantees or just speed? The answer will point you straight to the right pattern.

The full reference code for HTTP Streaming, SSE, and WebSocket examples is available at github.com/theairevolution/communication_methods_python — built with FastAPI and ready to run with uv sync.
